给 Coco AI 做 SEO：完整决策手册

0. TL;DR(60 秒读完)

我们为 Coco AI(coco.xyz)生产了 14 个 SEO 落地页,部署在 https://coco-seo.vercel.app/cases/。

  • 结构:10 个真实客户场景页 + 4 个产品/理念页
  • 方法论参考:高手网站 SEO 分析库(seo-wiki)、Simular.ai 三层递进模板、Coco 联创 Charlie 直接确认的案例边界
  • 核心交付物:coco-seo/ 项目(数据驱动 + 模板引擎 + 一键 Vercel 部署)
  • 总开发耗时:约 3 小时(含两次方向性返工)
  • 可复用度:技术骨架可直接套到下一个客户(改 data/pages.js + build.js 的品牌字符串即可)

最重要的一条经验:你以为你了解客户,你大概率不了解。在动笔之前,把"产品定位"从客户嘴里逼出来,而不是从会议记录里推断。 我在这次任务里因为没做这一步,返工了一次。

2. 第一次走偏:基于会议记录的"伪推断"

这一节我把自己犯的错完整记下来。对团队最有教育意义的不是我做对的部分,是我做错的部分。

2.1 我当时手里有什么

  • cases_coco.json:基于过去 8 次会议整理的 3 个案例(SOP 化)

  • qveris-seo 项目模板(用 qveris.ai 做 50 页 pSEO 的项目)

2.2 我做的(错误)推断

我把 "更安全更高效的 Claude Code" 当成 Coco 的当前产品定位,然后从这个定位往外推 10 个 SEO 页面:

  • Claude Code Alternatives for Enterprise Teams
  • How to Solve AI Agent Memory Loss in Long Coding Sessions
  • How to Make AI Coding Assistants Safe for Enterprise Use
  • ……(全是开发者工具角度)

这套方向逻辑自洽,但和客户实际定位差了一个时代。

2.3 哪里错了

用户给我发了 https://docs.coco.xyz/use-cases/ 后,我去抓取了实际页面,发现 Coco 当前的真实定位是:

维度 我以为 实际
定位 "更安全的 Claude Code" 企业级多 Agent 协作平台(Multi-Agent Coordination Platform)
主要受众 开发者 企业 CEO / 业务一把手
核心场景 写代码 客服/CRM/合规/财务/HR/投研/银行尽调
核心产品 Claude Code 增强 Zylos(开源 Agent 运行时) + HxA Connect(跨平台协作) + Coco Agent Cloud(云服务)
已验证案例 推断 200+ 付费企业,1001+ 生产案例

为什么会犯这个错:

  1. 会议记录里的定位语 ≠ 当前对外定位。"更安全的 Claude Code" 是几个月前的 A/B 测试结果,但产品已经迭代,对外定位已经升级到平台级。
  2. 没有看客户当前对外口径。客户的官网、docs、最新 pitch deck 才是权威定位源,会议记录是历史快照。
  3. "工程师做 SEO" 的偏差。我下意识把客户当成开发者工具公司(因为提到 Claude Code、Zylos),没有意识到 Coco 的核心客户其实是业务侧的 CEO/CFO/合规总监。

2.4 教训写进 SOP

客户 SEO 项目第一步:扒 ta 的当前对外口径,不要从内部会议记录推断

必扒清单: 1. 客户官网首页 H1 + 产品页 + Pricing 页 2. 客户 docs 站(如果有) 3. 客户最新一期 pitch deck / 一句话定义文档 4. 联创的最近一期公开访谈/直播(过去 30 天内)

直到客户能用一句话告诉你"我是 X for Y who do Z"之前,不要写一行 SEO 文案。

4. 方法论:为什么这样做 SEO

4.1 参考的两个核心框架

A. 高手网站 SEO 分析库(seo-wiki)

这是用户内部的另一个项目,沉淀了 60+ 篇高手 SEO 实战文章和 6 个核心策略(找词选品、建站上线、SEO 优化、外链建设、流量增长、变现复盘)。

我从中拿走的关键原则:

原则 应用
新词新站策略:找过去 12 个月才出现显著搜索量的词,竞争空白,新站可"上线即排名" 14 个页面全部围绕 2024-2026 才起量的 AI Agent 长尾词
E-E-A-T 四维:Experience(第一手经验)、Expertise(专业)、Authoritativeness(权威)、Trustworthiness(可信) 每页加一段 "First-hand reference" 模块,引用真实客户/数字
TDK 规范:Title 50-70 字符、Description 140-158 字符、关键词放最前 模板严格 enforce,build 时自动 clamp
结构化数据:HowTo + FAQPage + BreadcrumbList + SoftwareApplication 四套 schema 每页同时插入 4 个 JSON-LD
单 H1 + H2 分区:H1 每页只有 1 个,H2 是 section,H3 是 sub-item 模板硬保证
内容集群:父页 + 子页 + 内链网络 14 页两两互链,index 页按分类聚合
PageSpeed:目标 95+ inline CSS、无外部 JS、字体不外链

B. Simular.ai 三层递进模板(qveris-seo 沿用)

每个落地页按"递进式说服"结构展开:

  1. Pain(用户的具体痛苦,带数字)
  2. The Problem(痛苦的本质原因)
  3. Method 1: 手动 / DIY(承认现状方案,列优劣)
  4. Method 2: 现有工具(承认竞品,列优劣)
  5. Method 3: 客户产品(在最后 1/3 才出现)
  6. How to set it up(三步操作)
  7. FAQ(5 个真实问题)

关键:前 2/3 内容不提及客户产品。这是建立信任的核心动作——开发者/B 端读者最讨厌的就是"全篇都在卖你的东西"。先帮他理解问题,再给他选项,最后才说"我们也是一个选项,而且更好"。

4.2 为什么选这两个框架而不是别的

  • 不选 HubSpot 风格内容营销(品牌故事 + 漂亮配图):太软,B 端开发者不买账
  • 不选纯 KGR(关键词难度比)堆量:对客户品牌伤害大,内容质量太差
  • 不选 AI 一键生成 100 页:失控,质量不可控,被 Google 判低质风险

Simular.ai 的递进结构 + 高手库的 E-E-A-T 落实,是当前对 APAC B 端 SaaS 最匹配的组合。

5. 关键决策表

每一个决策都有 trade-off,这一节把"为什么这样选"摆出来。

5.1 站点定位:做 14 个独立长落地页,而不是博客流

选项 优点 缺点 选/不选
博客流(WordPress 风格) 容易扩展,RSS 友好 文章流到第 3 页就死了,SEO 价值低 ❌
14 个独立深度页(pSEO) 每页独立可索引,关键词聚焦 维护单页时间长 ✅
文档站(Mintlify 风格) 适合产品文档 不适合营销转化页 ❌

选 pSEO 的核心理由:每页都是一个完整的"问题-方案"叙事,可以独立做外链建设、独立做社交分发、独立追排名。

5.2 域名策略:coco-seo.vercel.app vs coco.xyz/cases/

我们目前部署在 coco-seo.vercel.app 子域。最终应迁移到 coco.xyz/blog/ 或 coco.xyz/cases/ 一级路径。

选项 优点 缺点
子域 seo.coco.xyz 部署独立 Google 判定为不同站点,主站权重不传导
一级路径 coco.xyz/cases/ 主站权重传导,DR 共享 需要客户配合做 reverse proxy 或集成到主站
独立站 cocoseo.com 完全独立 没有主站权重,从 0 起步

当前 coco-seo.vercel.app 是临时方案,等客户审核完内容后,谈一级路径迁移。

5.3 语言策略:英文优先,中文待定

Coco 国内版/国际版分开运营。我们这一期做的是国际版的英文 SEO。

理由: - 联创 Charlie 在新加坡,主战场是 APAC + 欧美 - Coco 海外定价用 USD/USDT,目标客户英文读写无障碍 - 中文 SEO 池子小,APAC 英文池子大且竞争未饱和

下一期:如果国内版需要 SEO,会用同一套数据驱动模板生产中文页,关键词替换为百度/Bing CN 的高搜索量词。

5.4 内容长度:每页 2500-3500 词,而不是 800 词

Google 在 2024 年后明显倾向"全面回答"的长内容,但代价是写作成本高。

长度 写作时间 排名表现 选/不选
800 词(博客平均) 1 小时/篇 长尾关键词难突破 ❌
1500 词 2 小时/篇 中等表现 ❌
2500-3500 词 4 小时/篇 高 Featured Snippet 几率 ✅
5000+ 词 8 小时/篇 边际收益递减,跳出率高 ❌

我们选 2500-3500 词,每篇含 5 个 sections + 3 steps + 5 FAQ + 1 CTA + 1 E-E-A-T 模块,字数自然落在区间内。

5.5 是否用 AI 生成内容

用,但不是一键生成。

  • 数据(每页的 before/after、sections 内容)由我手写,基于用户提供的 Q&A 手册和联创口径
  • 模板(HTML/CSS/Schema)由代码生成
  • 这是 "AI 辅助 + 人工把关" 模式,不是 "AI 一键 100 页" 模式

为什么不一键生成:

  1. AI 一键生成的内容没有客户口径(例如 Charlie 的原话引用、具体客户数字),E-E-A-T 直接归零
  2. AI 一键生成容易出现事实错误(把竞品名搞混、把客户案例编造),对客户品牌伤害大
  3. Google 在 2024-2025 多次明确"AI 内容本身不被惩罚,但低质 AI 内容会被惩罚",我们的赌注是质量,不是数量

6.2 关键词选择的两个原则

原则 1:不要追"AI" 头部词

像 "AI agent"、"AI for business" 这种头部词,DR 70+ 大站(Zapier、Hubspot、Forbes)已经占满。新站打不进去。

原则 2:追"AI + [角色] + [行业]" 的三段长尾

例如: - ❌ "ai customer service" (太宽,Intercom/Zendesk 占据) - ✅ "ai customer service agent restaurant chain franchise" (具体到行业)

每个页面的 keyword 都遵守这个三段式。这也是 Coco 自己 docs/use-cases 的"Role × Task × Industry"三维分类的延伸。

7. 技术实现

7.1 项目结构

coco-seo/
├── data/
│   └── pages.js          # 14 页内容数据(全部手写)
├── public/               # 构建产物
│   ├── index.html        # 首页(=cases/index.html)
│   ├── cases/
│   │   ├── index.html    # 资源中心,4 组分类
│   │   └── [slug]/index.html × 14
│   ├── sitemap.xml
│   └── robots.txt
├── build.js              # 模板引擎
├── package.json
├── vercel.json           # 部署配置
└── SEO-PLAYBOOK.md       # 本文档

7.2 build.js 做了什么

  1. 读 data/pages.js(14 页对象数组)
  2. 对每页:
  3. 生成 4 套 JSON-LD schema
  4. 自动 clamp meta description 到 140-158 字符
  5. 自动选 4 个 related pages(基于数组距离)
  6. 注入 E-E-A-T 段(从 eeatFor() 映射拿)
  7. 生成首页(按 4 组分类聚合)
  8. 生成 sitemap.xml(14 + index + root)
  9. 生成 robots.txt

核心设计原则:数据和模板分离。换客户只需要改 data/pages.js + 5 处品牌字符串(BRAND, TAGLINE, SITE 等)。

7.3 SEO 关键技术细节

点 实现
TDK title 强制 < 70 字符,description 自动 clamp 到 140-158
Canonical 每页绝对 URL,带 trailing slash(vercel.json trailingSlash: true)
Schema.org HowTo + FAQPage + BreadcrumbList + SoftwareApplication 四套
Open Graph + Twitter Card 每页都有
Sitemap XML,带 lastmod + priority
Robots 全开 + sitemap 引用
单 H1 模板硬保证
Inline CSS 无外部 CSS 文件,首屏 0 阻塞
无外链 JS 0 个 script(除 JSON-LD)
字体 系统字体栈,不引外部 webfont
移动适配 @media(max-width:640px) 单点断点
Crawl 预算 14 页全在 sitemap,内链 4×14=56 条相关链接,平均深度 ≤2

7.4 为什么不用 Next.js / Astro / 11ty

我们选了最简单的 Node.js 字符串模板。

理由: - 14 页静态站,Next.js 是大炮打蚊子 - 没有交互需求,React 是负担 - 客户随时可能让我们改 100 个细节(措辞、案例、数字),纯字符串模板改起来最快 - 部署到 Vercel 不需要任何 build framework,直接 node build.js 就出 HTML

下一期如果加搜索/筛选,再考虑迁移到 Astro 或 Eleventy。

8.2 Vercel 配置要点

  • public: true — 关键,确保 Deployment Protection 关闭(用户明确要求"去掉保护")
  • cleanUrls: true — /cases/zylos-xxx 而不是 /cases/zylos-xxx.html
  • trailingSlash: true — 统一 URL 形式,避免 canonical 重复
  • Cache-Control: public, max-age=3600, must-revalidate — Vercel CDN 缓存 1 小时,内容更新后 must-revalidate 拉新

10. 最容易踩的 5 个坑

按踩坑严重度从高到低排序。

坑 1:基于内部会议记录推断客户定位

症状:写了一周,客户说"这不是我们现在的定位"。

预防:Day 0 必须拿到客户当前对外口径(官网 + docs + pitch deck + 30 天内访谈),会议记录只做背景参考。

坑 2:案例提名没经过联创审核

症状:写完上线了,客户说"这个客户名我们不能提",紧急下线。

预防:Day 1 选题阶段就把"哪些客户可以提名 / 哪些可以提名但不写细节 / 哪些必须脱敏"列清单,联创签字才能写。

坑 3:Vercel Deployment Protection 没关

症状:部署完发链接给客户,客户看到 SSO 登录页,以为我们没部署成功。

预防:vercel.json 加 "public": true,部署后立刻 curl 一遍验证 HTTP 200(不是 401)。

坑 4:title/description 没 clamp 长度

症状:Google 把 title 截断成 "How to Build a Multi-...",description 显示一段后省略号。点击率掉一半。

预防:模板里硬 clamp。title 控制在 50-70 字符,description 控制在 140-158。本项目的 clamp() 函数已经做了。

坑 5:内容长度堆够了但没 E-E-A-T

症状:3000 字的内容,但全是泛泛而谈,没有真实数字、没有客户名、没有联创原话。Google 判低质,排名上不去。

预防:每页强制一个 "First-hand reference" 模块,写不出来就改写到能写出来为止。E-E-A-T 是 2024+ Google 排名的核心因子。

12. 给团队的行动指南

12.1 你接到一个 SEO 项目,先做这三件事

  1. 不要直接写。先扒客户当前对外口径,2 小时。
  2. 不要从零开始。先看 coco-seo/ 模板和 qveris-seo/ 模板,选一个改。
  3. 不要跳过联创审核。每一页落地之前,联创点头才能上线。

12.2 你想优化已有 SEO 项目,看这三个指标

  1. GSC 收录率:< 80% 说明技术 SEO 有问题(canonical / sitemap / robots)
  2. 平均 Position:停滞在 30-50 说明内容质量不够,加 E-E-A-T
  3. CTR:Position 高但 CTR 低,说明 title/description 没写好,优化 TDK

12.3 你接到客户的"内容修改"请求,这样响应

  1. 客户说"这段措辞改一下" → 改 data/pages.js,node build.js,vercel --prod,5 分钟交付
  2. 客户说"这个案例不能用了" → 改 data/pages.js + eeatFor() 映射,15 分钟交付
  3. 客户说"加一个新场景" → 复制现有页结构,写新数据,30 分钟交付
  4. 客户说"换个分组方式" → 改 build.js 的 GROUPS 数组,10 分钟交付

所有修改都是数据修改,不是代码修改。这就是模板引擎设计的意义。

12.4 你不确定怎么做的时候,问三个问题

  1. 客户的当前对外口径是什么? 找最新一份对外材料。
  2. 联创对这个细节怎么想? 直接问,不要猜。
  3. seo-wiki 里有相关案例吗? 60+ 篇高手实战,大概率有人踩过同样的坑。
