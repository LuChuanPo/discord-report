给 QVeris 做 SEO：完整决策日志

文档对象：AIWatch 团队成员（孔维彤、刘玉晗、孔月、张志博、程果） 目的：让你完整理解从零到一给一个客户做 SEO 的全过程——我做了哪些假设、哪些推断、哪些步骤、踩过哪些坑、最终沉淀了什么 SOP 写作时间：2026-04-19 核心原则：这个文档的价值不是"看完就知道怎么做 SEO"，而是让你看完以后，遇到下一个客户时能复用这套思考框架。SEO 不是套模板，是一系列连续的判断。我希望你看完能学会判断，而不是学会复制。

一、案例背景：QVeris 是谁，为什么这是个好的实战案例

QVeris 是出海 to B 客户，2026 年 4 月 3 日启动 Onboarding，目前合作 3 周，第一次 W1 阶段。

产品定位：AI capability routing 平台。用一句话讲就是"给 AI Agent 提供金融数据的中央调度层"——开发者通过 QVeris 的一个 API，可以调用 564 个评估过的金融数据 provider（Polygon、Finnhub、Alpha Vantage 等），覆盖 141 个金融能力（实时行情、公司基本面、宏观指标等）。

客户对接人：曲东奇（合伙人/运营），CEO 王林芳负责技术和融资。

为什么从我们的角度看，这是个有价值的训练案例：

  1. 典型的出海 to B 冷启动：种子轮、团队 10-15 人、CEO 没去过硅谷、零品牌认知、需要从零打造海外 SEO 资产
  2. 产品定位还在演化：从"API 聚合/工具路由"在向"Computer Use 基础设施"或"金融数据 MCP Server"演化——SEO 选词必须配合产品方向
  3. 赛道交叉：QVeris 横跨 AI Gateway、MCP 生态、金融数据 API 三个赛道，每个赛道的玩法都不一样——这正好让我们能演练"如何在交叉赛道找定位"
  4. 可量化验证：金融数据 API 是 B2B 高客单产品，KPI 清晰（注册数、API 调用量），SEO 效果可追踪

更重要的是，这是一个我们真实在踩坑的案例。下面记录的不是"我教你怎么做对"，而是"我做了 4 周，前 3 周方向都错了，第 4 周才找到真正能跑通的路径"。

二、踩过的坑（按时间顺序的完整事件链）

阶段 1：错误假设期（Week 1）—— 我做错的第一步是没验证

当时我在想什么：

QVeris 是 AI Routing 平台，对吧？那做 SEO 应该围绕这些词：AI gateway, AI router, multi-agent orchestration, computer use, claude code, MCP server。我让 Claude 帮我推一份 50 页的 how-to 教程列表，覆盖 5 个分类（AI Model Routing / Computer Use / Agent Orchestration / Comparison / Developer Tutorials）。

我怎么做的：

第一周我做了下面这些事，全部基于"假设"而非"验证"：

  1. 让 Claude 生成 50 个 how-to 标题（"How to Route AI Requests to the Best Model Automatically"……）
  2. 用 Claude API 批量生成每页 2500 字的内容，每页都包含 Method 1（DIY）+ Method 2（现有工具）+ Method 3（QVeris）的三层结构
  3. 用 SkillBoss API 给每页生成一张 hero 图
  4. 部署到 seo-doctor.vercel.app（一个独立 Vercel 子域名）
  5. 同时还做了 10 个 Capability 页面（基于 QVeris 自己的能力地图数据）

60 个页面，5 天上线。

我那时候还挺得意。心想"批量生产+真实数据嵌入+三层递进结构，应该是高质量 pSEO 了"。

这第一周的根本错误是什么：

我没有验证任何一个关键词的真实搜索量。我只是凭直觉觉得"AI gateway 这个词应该有人搜，因为 AI gateway 是个真实概念"。

但是——"AI gateway"和"How to Route AI Requests to the Best Model Automatically"是两回事。后者是我编出来的句子，可能根本没人搜。

阶段 2：发现问题——大罗 SEO 的视角

第一周做完后，曲东奇方向没改我也没新需求，我就回头审视自己的产出。这时候我看到大罗 SEO（@daluoseo）发了一条 Twitter，标题是《26 年发布的 AI SEO 工具 却还在用 22 年的优化逻辑》。他批评的是 Replit 的 AI SEO 功能，但他指出的"4 个过时假设"，我全部踩中了：

  1. ❌ 发得越多流量越大 —— 我一周上了 60 页
  2. ❌ 看起来像 SEO 就能排 —— 我做了 H1/H2/FAQ/Schema 全套
  3. ❌ 规模可以弥补信息增量不足 —— 我用 Claude 生成内容
  4. ❌ AI 能自动替你完成 SEO 判断 —— 我让 Claude 选词

我用一个简单脚本检查了一下我做的 50 个页面的相似度：

# 检查 Method 2 标题重复
17 个页面用了完全相同的 Method 2 标题：
"Method 2: Existing Tools (OpenRouter / LiteLLM / etc)"

# 检查 Method 3 标题重复
23 个页面用了完全相同的 Method 3 标题：
"Method 3: QVeris Intelligent Routing"

# 检查 CTA 文案
50 个页面的 CTA 文案完全一样（只有 1 种）

这就是大罗说的"工业化改写"——披着 AI 的皮的传统站群打法。Google 一眼就能识别。

学到的教训 1：每页必须有独特价值，不只是 H1 不同。Method 标题、CTA、内容结构、案例都必须不同。否则 Google 会把整批页面识别为低质量复制内容池，整体降权。

阶段 4：扩展关键词研究——找到真正的金矿

OK，既然我自己想的词全错，那就去看真实数据告诉我什么词有量。

我做了几件事：

4.1 围绕 QVeris 的产品赛道做种子词扩展

我让 Claude 列出 QVeris 真实涉及的所有赛道： - AI Gateway / LLM 路由（核心产品） - 金融数据 API（capability map） - AI 代理 / Computer Use / MCP（生态依赖） - 通用自动化竞品（横向参照）

然后每个赛道列 25-30 个种子词，全部丢进 DataForSEO 跑搜索量。

结果：627 个关键词，603 个有搜索量，98 个月搜 100+。

学到的教训 3：客户的产品定位经常在演化，SEO 选词必须基于客户产品当前真实支持的能力，不是你脑子里对客户的印象。每接手一个新客户，先把官网每个菜单都点一遍，看清楚他们到底支持什么。

4.3 但搜索量大≠值得做（KGR 的引入）

到这一步我又掉进了第二个陷阱。我看到 mcp server 有 60,500/月搜索量，立刻兴奋起来——这词 QVeris 一定要做！

幸好这时候我看到饼干哥哥的 SEO Dashboard 提到了 KGR（Keyword Golden Ratio）：

KGR = intitle 结果数 ÷ 月搜索量

KGR < 0.25 → 🟢极好（新站 3 个月内能排上去）
KGR 0.25-1.0 → 🟡一般（需要好内容+少量外链）
KGR > 1.0 → 🔴困难（新站不要碰）

KGR 衡量的是"竞争度"——已经有多少页面在抢这个词。intitle 结果数就是 Google 上有多少页面的标题包含这个词的精确匹配。

我把 30 个最看好的关键词跑了 KGR：

🔴困难 mcp server          60,500 vol  1,090,000 intitle  KGR 18.02
🔴困难 claude code skills   8,100 vol     30,300 intitle  KGR  3.74
🔴困难 ai trading bot       6,600 vol     48,600 intitle  KGR  7.36
🔴困难 financial data api   1,300 vol     15,500 intitle  KGR 11.92
🔴困难 mcp tools             2,400 vol    190,000 intitle  KGR 79.17

🟢极好 polygon pricing       2,900 vol           6 intitle  KGR  0.00
🟢极好 mcp server list       1,600 vol         274 intitle  KGR  0.17
🟢极好 openrouter alternative  480 vol          93 intitle  KGR  0.19
🟢极好 alpha vantage pricing   480 vol           7 intitle  KGR  0.01
🟢极好 stock api free          390 vol           5 intitle  KGR  0.01

Polygon Pricing 这个词，月搜索量 2,900，但 intitle 结果只有 6 个！ 这意味着虽然有人搜，但几乎没人写过深度的 Polygon 定价分析页面。这是新站能直接排到前 10 的金矿。

而 mcp server 这个 60,500 的大词，intitle 是 109 万——任何新站都不可能在这个词上排进前 10 名，3 年内都不可能。

学到的教训 4：搜索量和 KGR 必须双重验证。光看搜索量会被诱惑去抢大词，结果几年都排不上去；光看 KGR 会找到无人竞争但也无人搜索的死词。两者都要 OK 才值得做。

而我之前规划的"50 个 how-to + 10 capability"中，能通过 KGR 检验的只有 2-3 个。

阶段 5：从头制定计划——8 周渐进式

到这一步，我已经看清楚正确的路径了。我把整个 8 周计划重写：

Week 1: 10 页绿色 KGR 词验证期
   ├── 全部选 KGR < 0.25 的词
   ├── 每页都嵌入 QVeris 真实数据（564 providers 实测）
   └── 成功标准：7 天收录率 > 50%

Week 2: 10 页黄色 KGR 词扩展期
   ├── KGR 0.25-1.0
   └── 前提：Week 1 收录率 > 50% 才执行

Week 3: 30 页 Provider 定价矩阵
   ├── 每个 provider 一个定价页（KGR 普遍极低）
   └── 信息增量：QVeris 实测延迟和成功率

Week 4: 30 页 Alternative 矩阵
   ├── 直接竞品 alternative 页（高购买意图）
   └── X vs Y 对比页

Week 5-8: 根据前 4 周表现决定扩展方向

核心原则： 1. 渐进式发布——Week 1 上 10 页验证，确认能收录再扩 2. 每周固定的失败信号检查（收录率<50% 就暂停） 3. 信息增量必须来自 QVeris 真实数据，不是 Claude 编

学到的教训 5：所有 SEO 计划必须有"失败退出"机制。不是"我要做 280 页"，而是"我先做 10 页，2-3 个月后看数据，再决定是扩大、转向、还是撤退"。Type 1 决策（不可逆的，比如定位）必须慎重；Type 2 决策（可逆的，比如某一页内容）应该快速行动。

三、完整的决策框架（这是给客户做 SEO 时你应该用的）

把上面的踩坑过程压缩成可复用的框架，就是我们的 SEO 五层 SOP：

第 1 层：客户产品诊断（Decision Layer）

目标：用 1-2 小时把客户产品搞清楚，输出"客户 SEO 战略一页纸"。

10 个标准化问题（每个新客户都要回答）：

# 维度 选项
1 产品类型 工具型 SaaS / 内容型 / 平台型 / 数据型 / 服务型
2 主要用户 开发者 / 业务/营销 / 消费者(2C) / 企业采购 / 多角色
3 付费阈值 免费 / <$100/月 / >$1K/月 / 企业单 / 交易分成
4 主要搜索意图 信息型(how-to) / 对比型(vs) / 交易型(buy) / 导航型(品牌)
5 赛道阶段 蓝海新赛道 / 红海大厂混战 / 有明确竞品 / 品牌词为主
6 独家信息增量 这个产品有什么数据/案例/算法/集成是竞品没有的？
7 产品体验门槛 直接用(无注册) / 需要注册 / 需要对接/API Key
8 地域/语言 英文为主 / 中文为主 / 中英双语 / 多语言
9 产品迭代速度 周级(快) / 月级 / 季度级(慢)
10 流量期望时间 3 个月见效 / 6 个月 / 1 年长线

重点是第 6 个：独家信息增量。如果一个客户没有任何独家数据，SEO 是做不动的。Google 现在不奖励"重新组织已有信息"的页面。

QVeris 的独家信息增量是：564 个金融数据 provider 的实测延迟、成功率、价格数据。这是别人没有的，所以我们做的每页对比都比竞品对比文有信息增量。

但如果你接到一个客户，发现他们没有任何独家数据——你必须先帮他们积累信息增量，再做 SEO。否则就是在水池里浇水。

第 2 层：关键词研究（Execution Layer，工具化）

工具：DataForSEO API（$5/月够用）

步骤：

# Step 1: 列出种子词
# 围绕客户产品的 4-6 个赛道，每个赛道 25-30 个词
# 包括：核心功能词、竞品品牌词、长尾应用场景词

# Step 2: 批量查搜索量
POST https://api.dataforseo.com/v3/keywords_data/google_ads/search_volume/live
[{"keywords": [...], "language_code": "en", "location_code": 2840}]

# Step 3: 过滤
volumes.filter(v => v.volume > 0)  # 没搜索量的直接 pass

# Step 4: 批量跑 KGR
for keyword in volumes:
    intitle = SERP_API(f'intitle:"{keyword}"').count
    kgr = intitle / volume

# Step 5: 评级
kgr < 0.25 → 🟢 优先做
kgr 0.25-1.0 → 🟡 配合外链才做
kgr > 1.0 → 🔴 不做

项目里已有的脚本： - verify-keywords.js —— 批量查搜索量 - kgr-check.js —— 批量跑 KGR - 脚本路径：C:/Users/langh/OneDrive/Desktop/claude code/qveris-seo/

给团队的硬规则：每个客户的 SEO 计划开始前，必须先跑 100+ 关键词的 DataForSEO 验证。不允许用直觉选词。这是不可妥协的红线。

第 3 层：定位决策（Decision Layer，需要你/我判断）

这是唯一不能完全 scale 的环节。基于 Layer 1+2 的数据，做出"做哪类页面、不做哪类"的决策。

决策规则（从 QVeris 项目沉淀的）：

IF 搜索意图 == "交易型(buy)" AND KGR < 0.25:
    → 优先做 Alternative 页 / 对比页 / 定价页（购买意图最强）

ELIF 搜索意图 == "信息型(how-to)" AND KGR < 0.25 AND 搜索量 > 100:
    → 次优做 概念解释页 / 列表页

ELIF KGR 在 0.25-1.0 AND 搜索量 > 500:
    → 需要好内容+外链才做（不是优先级）

ELIF KGR > 1.0:
    → 直接 pass，不浪费时间

输出：每个客户一份"关键词执行优先级清单"，包括： - Top 10 必做关键词（KGR 极好） - Top 20 次要关键词（KGR 一般） - Pass 列表（KGR 困难，3 个月内不碰）

第 4 层：内容生产（半工具化）

模板可以共享，但每页的"独特信息增量"必须 custom。

模板部分（所有客户共用）： - HTML 结构 - Schema markup（FAQ、HowTo、SoftwareApplication） - 内链规则（每页 3-5 条相关内链） - CTA 框架

Custom 部分（每个客户必须自己产出）： - 独家数据（QVeris 是 564 provider 实测；其他客户必须有自己的独家数据） - 真实案例 - 客户视角的痛点描述（不是 AI 编的"通用痛点"） - CTA 文案（每类页面要不一样，不能 50 页都是"试用 QVeris"）

质量底线（来自 Kostja 的 4 条 + 大罗的补充）：

  1. ✅ 唯一的 Title + Meta（不能用同一个模板标题）
  2. ✅ 非通用的正文（每页必须有该页独有的数据）
  3. ✅ 可追溯的事实（数据有来源，能链接到原始资料）
  4. ✅ 明确的 CTA（不是信息死胡同）
  5. ✅ 前 400 词包含核心结论（AI 搜索只读 377 词）
  6. ✅ 至少 3 个具体统计数据（GEO 优化 +37% 可见性）
  7. ✅ 至少 1 个权威引用 + 出站链接（GEO 优化 +40% 可见性）

差异化硬指标：任意两个页面的相似度必须 < 30%（用 TF-IDF 或 embedding 计算）。Build 前要跑相似度检测脚本，> 30% 的页面阻断部署。

第 5 层：监控与优化（完全工具化）

每周监控：

指标 目标 不达标的处理
7 天收录率 > 60% 降速到 20 页/周，加大外链建设
crawled-not-indexed < 15% 检查内容质量，合并相似页面
KGR < 0.25 词的排名 2 周内进前 50 增加外链，优化 On-page

每月检查： - 有排名（前 100）的页面数 / 总页面数 → 应该 > 50% - 前 20 名关键词数 → 月增 20+ - 搜索流量 → 月增 300%+

关键决策点： - 每 2 周一次"扩量 / 暂停 / 转向"决策 - 决策必须基于数据，不是直觉

四、给团队的具体执行指导

4.1 当你拿到一个新客户，第一周做什么

Day 1（半天）：客户产品研究 - 把客户官网每个菜单都点一遍，记录他们真实支持什么 - 看客户的 Twitter/LinkedIn，了解定位和叙事 - 看客户的 GitHub 或 Docs（如果有），了解技术栈 - 重点：他们的"独家信息增量"是什么？把它写在一句话里

Day 1（半天）：Onboarding 会议 - 用 10 维诊断问卷过一遍 - 重点确认搜索意图和付费模式 - 让客户给 5 个他认为的"必做关键词"——之后我们会用 DataForSEO 验证

Day 2（一整天）：关键词研究 - 围绕客户的 4-6 个赛道列种子词（每赛道 25-30 个） - 跑 DataForSEO 搜索量 - 挑出有量的（>100） - 跑 KGR - 输出：Top 10 必做 + Top 20 次要 + 失败列表

Day 3（半天）：竞品分析 - 用 DataForSEO ranked_keywords API 拉取 5 个主要竞品的 SEO 关键词 - 看哪些词竞品已经在排名，哪些是空白 - 输出：竞品对照矩阵

Day 4（一整天）：定位决策 + 第一周计划 - 跟我（郎瀚威）或资深成员一起做 Layer 3 决策 - 决定第一周做哪 10 页 - 准备每页的独特信息增量来源

Day 5（一整天）：第一周内容生产 - 用 Claude API 批量生成 - 跑相似度检测 - 部署 + 提交 GSC

4.2 内容生产的具体规则

❌ 不要做的事：

  1. 不要让 Claude 帮你"想"关键词——它会编出听起来像但没人搜的句子
  2. 不要批量用同一个 prompt 生成 50 页——内容会高度雷同
  3. 不要用 Claude 生成"通用痛点描述"——Google 能识别这种模板内容
  4. 不要在没确认 KGR 的情况下做大词
  5. 不要一次性发布 50+ 页面——必须渐进式
  6. 不要跳过 GSC 提交——AI 写的内容如果不主动提交，可能永远不被收录
  7. 不要用同一个 CTA 文案在所有页面上——这是被识别为"批量内容"的信号

✅ 必须做的事：

  1. 每页必须有该页独有的数据/案例
  2. 每页前 400 词必须包含核心结论（不要把精华埋在第 3 段以后）
  3. 每页至少 3 个具体统计数据，至少 1 个权威引用
  4. 每页 H2/H3 结构要不同（不能 50 页都是 Method 1/2/3）
  5. 每页 CTA 文案要根据页面类型变化
  6. Build 前跑相似度检测，>30% 阻断
  7. 部署后立即 GSC 提交 sitemap + 手动请求索引前 5 个页面
  8. 部署后 24 小时内在 Reddit/HN 推广 2-3 个最重要页面（加速收录）

4.3 KGR 验证的执行细节

步骤：

  1. 拿到 50-100 个种子关键词，跑 DataForSEO 搜索量
  2. 过滤掉零搜索的
  3. 剩下的按搜索量降序排列
  4. 从前 30 个开始跑 KGR（用 SERP API 查 intitle:"keyword" 的结果数）
  5. 每个 KGR 查询要等 1.5 秒（rate limit）
  6. 记录到 CSV

KGR 解读：

KGR < 0.10  → 🟢💎 极少竞争（必做）
KGR 0.10-0.25 → 🟢 低竞争（必做）
KGR 0.25-0.5  → 🟡 中等竞争（要好内容）
KGR 0.5-1.0   → 🟡 较高竞争（需要外链配合）
KGR > 1.0     → 🔴 高竞争（新站不碰）
KGR > 5.0     → 🔴 极高竞争（除非有大量外链否则不可能）

特殊情况：

  • 搜索量很小但 KGR 也极小（比如搜索量 20，intitle 3）：这是 long-tail gold，可以做但不要花太多精力
  • 搜索量极大（10K+）但 KGR > 5：千万不要做。这是 trap——大词背后有 Wikipedia/官方文档/大厂博客在排名，新站完全没机会

4.4 失败信号的识别和处理

信号 1：3 个月后大部分词都没排进前 100 → 可能问题：定位错了 → 处理：回到 Layer 1 重新诊断，可能客户的真实用户和我们假设的不一样

信号 2：被 Google 收录但没排名 → 可能问题：KGR 估算错了，词的真实竞争比 KGR 显示的高 → 处理：跑前 10 名页面的 DR/外链分析，看是不是大厂占位

信号 3：有排名但点击率极低 → 可能问题：搜索意图错了，用户搜这个词其实想要别的 → 处理：重新看 SERP，看排名前 5 的页面是什么类型，调整内容方向

信号 4：有点击但没注册转化 → 可能问题：用户画像错了 → 处理：可能我们以为是开发者，其实是学生；或者付费意愿不在我们想的那个点上

每种失败都有明确的诊断和回滚机制。最坏情况只是浪费 10 页的工作（3-5 天），不是 6 个月。这就是 MVP 验证的价值。

六、错误案例库（不要再犯的错）

错误 1：批量生成关键词清单不验证

犯错场景：让 Claude 帮我列了 50 个 how-to 标题，没验证就开始写 结果：50 个全部零搜索 正确做法：所有关键词必须先过 DataForSEO 验证。没有数据支撑的关键词不允许进入内容生产环节。

错误 2：50 个页面用同一个 Method 模板

犯错场景：让 Claude 用同一个 prompt 生成 50 页，结果 17 个页面 Method 2 标题完全一样 结果：Google 会把整批页面识别为"工业化改写" 正确做法：Build 前跑相似度检测脚本，相似度 > 30% 阻断部署。或者每批 10 页用不同的 prompt 模板。

错误 3：50 个 CTA 文案完全一样

犯错场景：所有 50 个页面 CTA 都是"试用 QVeris 免费版" 结果：被 Google 识别为"低质量批量内容"的强信号 正确做法：根据页面类型差异化 CTA。对比页："看完整 564 provider 评测"；定价页："对比 QVeris 的实际成本"；教程页："立即试用 QVeris MCP Server"

错误 4：抢搜索量大但 KGR 极高的词

犯错场景：看到 mcp server 60,500 搜索量，立刻规划 Pillar Page 结果：intitle 109 万，新站 3 年都排不上去 正确做法：搜索量和 KGR 必须双重验证。新站只做 KGR < 0.25 的词。

错误 5：一次性上线 50+ 页面

犯错场景：一周做完 60 页，全部 deploy 结果：Google 爬虫预算分配不过来，大部分页面 crawled-not-indexed 正确做法：渐进式发布，Week 1 上 10 页验证，等收录率 > 60% 再扩大。

错误 6：忘记提交 GSC 和外链建设

犯错场景：60 页部署完就以为结束了 结果：30 天后只有 5 页被收录 正确做法：部署后必做： 1. GSC 提交 sitemap 2. 手动请求索引前 5 个最重要页面 3. 提交到 ProductHunt / G2 / AlternativeTo / Capterra（外链冷启动） 4. 在 Reddit / HN 发 2-3 个有价值的讨论帖（含页面链接）

错误 7：用 AI 写"通用痛点"

犯错场景：让 Claude 编写"开发者面对多 vendor 的痛苦" 结果：和其他 100 个 SEO 文章读起来一模一样 正确做法：痛点必须来自客户的真实用户访谈或客户内部数据。如果没有，就先去 Reddit/Twitter 看真实的用户吐槽，提炼出来。

七、给团队的最后建议

7.1 心态：SEO 是判断的艺术，不是执行的体力活

大罗的原话："AI 牛逼的地方是快速把行业公开经验整合，对小白入门到平均水平是神器。但想精进就会发现 AI 答案趋向于平庸。"

你的判断力比你的执行力值钱 10 倍。如果你只会用 Claude 批量生成内容，你的工作 6 个月内会被自动化淘汰。但如果你能做关键词战略判断、产品定位判断、信息增量判断——这些能力 5 年都不会贬值。

7.2 学习路径

第一周： - 把这份文档完整读一遍 - 把 SEO Dashboard (seo-dashboard-plum.vercel.app) 的全部内容读一遍 - 把 QVeris 项目的脚本代码看一遍（重点看 verify-keywords.js 和 kgr-check.js）

第二周： - 选一个不重要的小客户，用我们的 SOP 跑一遍（从诊断到 Week 1 计划） - 跑完和我对一次，看你的判断和我的判断差距在哪 - 修正你的认知

第三周： - 独立完成一个客户的诊断+关键词研究+Week 1 计划 - 我只做最终 review，不参与决策

第四周： - 你能独立做完整的 SEO 项目 - 开始指导别的同事

7.3 红线规则（不允许触碰）

  1. ❌ 不允许跳过 DataForSEO 关键词验证——任何关键词都必须有数据支撑
  2. ❌ 不允许跳过 KGR 检验——只做 KGR < 1.0 的词
  3. ❌ 不允许批量生成相似页面——任意两页相似度 > 30% 必须重写
  4. ❌ 不允许一次性上线 30+ 页面——必须渐进式
  5. ❌ 不允许用 AI 生成"通用痛点"——必须有真实数据/案例
  6. ❌ 不允许跳过 GSC 提交和外链建设——内容做完不等于结束

八、结束语：从 QVeris 项目我学到的最深一课

最深一课不是某个具体技巧，而是：

SEO 不是一次性项目，是一个持续的判断游戏。

我们一开始觉得"做 60 页就完了"，后来发现这 60 页全部走错路。但这不是失败——这是必经的学习过程。任何一个客户的第一批内容有 80% 都会被推翻重做。关键不是一开始就做对，而是：

  1. 快速做小样本（10 页验证）
  2. 快速看数据（2-4 周收录率）
  3. 快速调整方向（数据不达标立刻改）

不要害怕做错。害怕的应该是"做错了还在大规模执行"。

我们这次踩的所有坑，都不是因为我们笨——而是因为我们没用真实数据驱动决策。一旦接入了 DataForSEO 和 KGR 这套工具链，前面所有的"凭直觉"都被替换成了"凭数据"。然后所有的判断都开始变得明确。

这就是为什么我说 SEO 是判断的艺术：不是你比别人勤奋，是你比别人更早地用数据替换了直觉。

希望你看完这份文档后，下一次接客户时不需要踩同样的坑。我们的 SOP 就是这些坑的浓缩。

附录 A：常用 DataForSEO API 调用模板

A.1 批量查搜索量

curl -X POST "https://api.dataforseo.com/v3/keywords_data/google_ads/search_volume/live" \
  -H "Authorization: Basic <BASE64>" \
  -H "Content-Type: application/json" \
  -d '[{
    "keywords": ["keyword1", "keyword2", ...],
    "language_code": "en",
    "location_code": 2840
  }]'

每次请求 $0.075，可一次查 700 个关键词。

A.2 跑 KGR

curl -X POST "https://api.dataforseo.com/v3/serp/google/organic/live/advanced" \
  -H "Authorization: Basic <BASE64>" \
  -H "Content-Type: application/json" \
  -d '[{
    "keyword": "intitle:\"your keyword\"",
    "location_code": 2840,
    "language_code": "en",
    "device": "desktop",
    "depth": 10
  }]'

每次请求 $0.013，可读取 result[0].se_results_count 作为 intitle 数。

A.3 拉竞品排名词

curl -X POST "https://api.dataforseo.com/v3/dataforseo_labs/google/ranked_keywords/live" \
  -H "Authorization: Basic <BASE64>" \
  -H "Content-Type: application/json" \
  -d '[{
    "target": "competitor.com",
    "language_code": "en",
    "location_code": 2840,
    "limit": 100,
    "order_by": ["keyword_data.keyword_info.search_volume,desc"]
  }]'

每次 $0.013，看竞品在哪些词上排名。
